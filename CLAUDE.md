# Dune Experiments

**Автор:** K1r1llKo  
**Тип:** Браузерная RTS — ремейк Dune 2 с улучшенным балансом, аутентичной атмосферой и современной графикой.  
**Платформа:** Desktop браузер (Chrome/Firefox/Safari). Мобильная версия не планируется.

---

## Стек

| Слой | Технология | Почему |
|------|-----------|--------|
| Язык | TypeScript | Строгие типы = меньше багов при расширении |
| Рендер | PixiJS v7 | WebGL, батчинг спрайтов, поддержка RenderTexture |
| Сборка | Vite | Мгновенный HMR, zero-config TS, простой деплой |
| Стиль | CSS (без фреймворка) | Минимальный UI, Courier New пиксельная эстетика |

**Запуск:**
```bash
npm install
npm run dev     # dev-сервер → http://localhost:5173
npm run build   # сборка в dist/
```

## Структура проекта

```
src/
  types/        — TypeScript интерфейсы (Unit, Building, Worm, GameState и т.д.)
  constants/    — Числа и таблицы: юниты, здания, фракции, размер карты
  state/        — Глобальный стейт игры (создаётся в startGame, живёт до endGame)
  map/          — Генерация карты, типы тайлов, проходимость
  entities/     — Фабрики юнитов/зданий/снарядов, без логики обновления
  systems/      — Вся игровая логика (FSM комбайна, туман, AI, червь, ремонт и т.д.)
  render/       — PixiJS: слои сцены, спрайты, миникарта
  ui/           — DOM: HUD, сайдбар, меню, тосты
  input/        — Мышь, клавиатура, камера
  main.ts       — Точка входа, PIXI.Application, тикер
assets/
  sounds/       — WAV/OGG (будущее)
```

## Архитектурные решения

### Стейт — плоский объект
`GameState` — один объект с массивами `units[]`, `buildings[]`, `projectiles[]`, `worms[]`, `fx[]` и плоскими полями.
**Никаких классов с наследованием.** Каждый объект — простой `interface`, создаётся фабрикой (`makeUnit`, `makeBuilding`).
Это позволяет легко сохранять/загружать стейт и делать скирмиш с несколькими играми.

### Системы — чистые функции
Каждый файл `systems/*.ts` экспортирует чистые функции вида:
```ts
export function updateHarvester(unit: Unit, dt: number, state: GameState): void
```
Никакого глобального мутабельного стейта вне `GameState`.

### Рендер — отделён от логики
`render/renderer.ts` только читает `GameState`, никогда не пишет.  
Синхронизация `GameState → PixiJS объекты` происходит через `syncScene(state)`.

### Туман войны
3 состояния на тайл: `0=unexplored | 1=explored | 2=visible`.  
Пересчитывается каждый тик через `updateFog(state)`.  
Рендеритется слоем `FogLayer` поверх карты.

### Производственная цепочка
Здание может производить юниты только если оно достроено и имеет питание.  
Тех-дерево описано в `constants/buildings.ts` через поле `prereq: BuildingKind[]`.

### Апгрейды зданий
Некоторые здания поддерживают апгрейд через виртуальные `BuildingKind`-токены (`heavyupg`, `yardupg`).  
`hasBuilding(faction, 'heavyupg', state)` не ищет отдельное здание — проверяет `b.upgraded === true` на `heavy`.  
Аналогично `'yardupg'` → `b.upgraded` на `yard`.  
Апгрейд стоит spice, занимает одну кнопку в BUILD-панели, выполняется мгновенно (`upgradeBuilding` / `aiUpgradeBuilding` в `systems/production.ts`).

### Поиск пути (A*)
`systems/pathfinding.ts:findPath(state, sx, sy, gx, gy)` — 8-направленный A* по сетке зданий.  
Путь кэшируется на юните (`u.path`), пересчитывается при смене цели и троттлится через `u.pathRecomputeIn`.  
**A* используется и для chase, и для move-order** — юниты огибают здания при любом приказе, не лезут «по диагонали через стену». Чтобы не пересчитывать каждый кадр для движущейся цели, расчёт ограничен раз в 0.6с (chase) / 1.5с (move-order). Хавестер тоже использует A* (`systems/harvester.ts:moveTo`).  
Если путь не найден за 4000 узлов — fallback на прямое движение с building repulsion в `directMove`.  
`findFreeSpot(state, x, y)` — ищет ближайший свободный тайл кольцами от точки (используется при спавне юнитов и выезде харвестера из дока).

### Док рефинерии
Хавестер не «сливает» спайс снаружи — он въезжает внутрь рефинерии. FSM:
1. `toRefinery`: путь к точке `(ref.tx + ref.w − 0.5, ref.ty + ref.h − 0.5)` — внутрь правого нижнего тайла
2. На дистанции < 1.0 от точки дока — телепорт в координаты дока, `unit.docked = true`, переход в `depositing`
3. Во время `depositing` — спрайт скрыт (рендер пропускает docked), юнит не учавствует в combat/separation/sandworm
4. По таймеру — выезд через `findFreeSpot` рядом с рефинерией, `docked = false`

Поле `Unit.docked: boolean` — глобальный флаг для всех систем фильтрации (`combat.ts`, `sandworm.ts`, `carryall.ts`, рендер).

### Энергобаланс
`powerOf(faction, state)` возвращает `{ prod, cons, ratio }`.  
`ratio < 1` замедляет производство до `30%` минимума и снижает скорость добычи.

### Авто-ремонт зданий
`systems/repair.ts` каждый тик восстанавливает HP всем повреждённым зданиям владельца, у которого жив Construction Yard. Стоимость пропорциональна `def.cost` (см. `REPAIR_RATE`, `REPAIR_COST_PER_HP` в `constants/buildings.ts`). Если у фракции не хватает spice — ремонт ставится на паузу.

### Spice blooms
`systems/spiceBloom.ts` каждые ~30с подсыпает новое spice-пятно (радиус 2) на случайный песчаный тайл вдали от баз. Без этого экономика «вырабатывалась» к 10-й минуте и игра застревала.

### Sandworm
`state.worms[]` — отдельный массив (не юниты). Червь имеет FSM `underground → travel → dive`. Спавнится только на песке и **охотится только на юнитов на песке** (rock — безопасная зона). Если жертва уходит на камень во время `travel`, червь теряет цель и уходит на dive.

Параметры выбора цели (`findVictim`):
- Базовый вес: 1.0
- Комбайн: ×0.4 (приоритетная цель)
- Docked-харвестер внутри рефинерии: пропускается полностью
- Юнит на камне: пропускается полностью

**Тайминги спавна:**
- Первый червь: **через 5 минут** после старта игры (`wormTimer = 300` в `createGame`). Это даёт обеим сторонам построить экономику без потери стартового харвестера.
- Последующие: каждые ~75с (`SPAWN_BASE_INTERVAL`), максимум 2 одновременно.

На приближении (`WORM_ATTACK_RANGE = 0.7`) — мгновенно убивает цель и уходит. Подробно — `systems/sandworm.ts`.

### Турели (статичные защитные здания)
Здания с полем `weapon: TurretWeapon | null` в `BUILD_DEFS` стреляют по ближайшему врагу в радиусе. `systems/combat.ts:updateTurrets` ведёт `cooldown`, `attackTarget` и направление ствола `dir` — рендер вращает отдельный спрайт-«ствол», прикреплённый к контейнеру здания.  
Спрайт ствола рисуется размером `TILE×TILE` с прозрачной заливкой на весь фон — это фиксирует anchor точку в центре тайла для корректного вращения вокруг основания башни.

### Танки с независимой башней
`Combat Tank` (`kind === 'tank'`) и `Siege Tank` (`kind === 'siegeTank'`) — мобильные юниты с раздельным корпусом и башней.
- `Unit.dir` — направление корпуса (по движению).
- `Unit.turretDir` — направление башни. Плавно (5 рад/с) поворачивается к `attTarget` при наличии цели; иначе возвращается к `u.dir`.
- В рендере у этих юнитов два child-спрайта: `'body'` (rotation = `u.dir`) и `'turret'` (rotation = `u.turretDir`).
- Спрайты башни: `unit_${faction}_tank_turret` (16×16), `unit_${faction}_siegeTank_turret` (24×24, два ствола). Прозрачный фон для корректного pivot.
- При стрельбе в радиусе `u.dir` НЕ снапается на цель — только башня поворачивается. Корпус сохраняет направление движения.

### Стелс-механика
Поле `UnitDef.stealthRange?: number` (тайлы). Если задано (>0), юнит невидим вражеским сторонам, кроме случая когда юнит/здание противника находится в радиусе `stealthRange`.

`isStealthVisible(state, target, viewerFaction)` в `combat.ts` — единая функция, используется в:
- `findNearestEnemy` (юниты не цепляются за невидимый стелс)
- `findNearestEnemyForBuilding` (турели тоже)
- `pickEntity` в `mouse.ts` (нельзя кликнуть)
- `renderer.ts` (спрайт скрыт)

Юниты со стелсом: **Saboteur** (4 тайла), **Stealth Tank** (3 тайла).

### Anti-air (`UnitDef.antiAir?: boolean`)
Юнит может атаковать воздушные цели (`fly: true`). По умолчанию false.  
Носители AA: `launcher`, `stealthTank` (новый, эксклюзив Ordos), позднее `Sardaukar`, `Fremen`.  
В `findNearestEnemy` летающие цели отфильтрованы для не-AA юнитов.

### Saboteur — kamikaze-механика
`kind === 'saboteur'` — особая ветка в главном цикле:
- Не имеет projectile, range=0.
- Auto-target ищет ТОЛЬКО вражеские здания в радиусе 12.
- ПКМ игроком по вражескому зданию → `attackTarget = building.id`.
- При сближении (rect-distance < 0.6) → `damageEntity(state, target, maxHp+1)` мгновенно убивает здание; юнит погибает, эффект взрыва.
- В `directMove` — building repulsion отключена для саботера (иначе застрянет в 1.1 тайле от цели).

### Faction-restricted юниты — `canFactionProduce(faction, kind)`
- `saboteur`, `stealthTank` — только Ordos
- `siegeTank` — только Atreides и Harkonnen

`findProducer` возвращает `null` если фракция не может строить юнит. Sidebar скрывает кнопку.

## Игровые фракции

| Фракция | Цвет | Спец-юнит | Особенность спец-юнита |
|---------|------|-----------|------------------------|
| Atreides | `#3a78d8` Синий | Sonic Tank | Звуковая волна, area damage (splash 1.4) |
| Harkonnen | `#cc2424` Красный | Devastator | Тяжёлая броня (520 HP), splash 1.8, медленный (speed 1.0) |
| Ordos | `#2c9c44` Зелёный | Saboteur | Kamikaze — уничтожает здание при контакте, гибнет сам. Stealth 2 тайла. |

Фракции фиксированы — три, как в оригинальной Dune II. Новые фракции не планируются.  
Новые **юниты и здания внутри фракций** — допустимы и приветствуются.

### Stats берутся через `statsFor(faction, kind)`
`UNIT_STATS` содержит базовые цифры для всех типов кроме `'special'` и `'trike'`.
- Для `'special'` функция `statsFor` подставляет значения из `SPECIAL_STATS[faction]`.
- Для `'trike'` — из `TRIKE_STATS[faction]` (Atreides Trike / Harkonnen Quad / Ordos Raider — три тактически разных машины).

Все игровые системы (combat, fog, production, sidebar) обязаны использовать `statsFor`, а не `UNIT_STATS[kind]` напрямую — иначе получите неверные цифры для трайка/спец-юнита.

### Философия фракционной разницы Trike-варианта
- **Atreides Trike** — сбалансированный baseline, универсальный.
- **Harkonnen Quad** — медленный, толстый, больно бьёт. Цена выше. Тяжёлый штурмовик-таран.
- **Ordos Raider** — хрупкий, но самый быстрый, с увеличенной дальностью и лучшим обзором. Дешевле и быстрее в производстве. Идеальный скаут / партизан.

### Faction Perks (`FACTION_PERKS` в `constants/factions.ts`)
Глобальные пассивные модификаторы, применяются поверх базовых статов в `statsFor()` (для юнитов) и `makeBuilding()` (для зданий) и в системных тиках (`power.ts`, `production.ts`, `harvester.ts`).

| Перк | Atreides | Harkonnen | Ordos |
|---|---|---|---|
| Building HP × | 1.0 | **1.2** | 1.0 |
| Carryall HP × | **1.5** | 1.0 | 1.0 |
| Unit HP × | 1.0 | 1.0 | **0.95** |
| Unit DMG × | 1.0 | **1.15** | 1.0 |
| Unit Speed × | 1.0 | **0.9** | **1.05** |
| Unit Cost × | 1.0 | 1.0 | **0.9** |
| Unit Range × | **1.1** | 1.0 | 1.0 |
| Unit Sight × | **1.1** | 1.0 | 1.0 |
| Wind Trap power | 8 | **10** | **12** |
| Refinery cap × | 1.0 | **1.5** | 1.0 |
| Refinery passive | — | — | — *(removed: создавал runaway snowball)* |

**Реализация:**
- `statsFor(faction, kind)` мерджит base stats × perk multiplier → системы (combat/fog/production/UI) получают modified значения автоматически.
- `makeBuilding()` применяет `buildingHpMul` к `def.hp` → каждое здание Харконнена строится с +20% HP.
- `powerOf(faction, state)` использует `windTrapPower` вместо фиксированного `+8` для всех Wind Trap.
- `harvester.ts` в `depositing` — потолок депозита = `state.cap * refineryCapMul`.
- `production.ts:updateProduction` поддерживает `refineryPassiveSpice` (early-out при 0); сейчас все фракции имеют 0 — Ordos потерял пассивный доход в патче баланса.

**Дизайн-паттерн:** перки **стэкуются** с фракционными overrides (TRIKE_STATS, SPECIAL_STATS). То есть Harkonnen Quad имеет DMG 15 (override) × 1.15 (perk) ≈ 17 итоговых.

## Производственная цепочка

```
Construction Yard ──[upgrade: yardupg 600]
    └── Wind Trap (+8 power)
            └── Refinery (−3 power)
                    ├── Barracks (−2) → Infantry
                    │       └── Radar Outpost (−4)
                    │               ├── Gun Turret (1×1, −2)              [защита от пехоты]
                    │               └── Rocket Turret (1×1, −3)   [требует yardupg]
                    └── Light Factory (−3) → Trike
                            └── Heavy Factory (−4) → Tank / Harvester ──[upgrade: heavyupg 500]
                                    │                                          └── Launcher
                                    └── Hi-Tech (−5) → Special / Carryall
                                            └── Palace (3×3, −6) [требует yardupg] → Faction superweapon
```

Победная цель: уничтожить Construction Yard противника.

### Турели
- **Gun Turret** — 250 spice, HP 450, range 4.5, dmg 18, cd 0.85с, `bullet`. Периметр против пехоты; требует Radar.
- **Rocket Turret** — 400 spice, HP 550, range 6.5, dmg 30, splash 0.6, cd 1.5с, `rocket`. Антитех-защита; требует апгрейд Construction Yard (`yardupg`).

Турели — обычные `Building` с `weapon: TurretWeapon | null`. `productionQueue` у них пуст. Боевые поля в `Building` (`cooldown`, `attackTarget`, `dir`) изначально нулевые и используются только если `def.weapon !== null`.

### Palace — апекс-здание с фракционной супер-способностью
**Palace** — 999 spice, HP 2000, 3×3, требует `hitech` + `yardupg`. Каждая фракция получает уникальную супер-способность с кулдауном `PALACE_COOLDOWN` (240с). Первая зарядка после постройки занимает 300с (поле `Building.superCooldown`).

| Фракция | Способность | Реализация |
|---|---|---|
| **Atreides** | **Fremen Squad** — 5 элитных воинов появляются на ближайшем к врагу краю карты и идут к Construction Yard противника. | `spawnFremenSquad` в `systems/palace.ts`. Юнит `'fremen'`: HP 100, DMG 14, range 3, speed 2.0, **stealth 2** (полу-скрытность), bullet. Не строится фабриками (`canFactionProduce('fremen')` всегда false). |
| **Harkonnen** | **Death Hand** — баллистическая ракета на любую точку карты с разбросом ±1.5 тайла. dmg 320, splash 5. | `launchDeathHand` спавнит `Projectile { kind:'deathHand' }`. Игрок входит в режим `palaceTargetingId`: следующий ЛКМ по карте — координата удара, ПКМ — отмена. |
| **Ordos** | **Free Saboteur** — спавнит бесплатного `'special'` (= Ordos Saboteur) рядом с Palace, направленного к вражескому CY. | `spawnFreeSaboteur` использует существующую kamikaze-логику Ordos special из `main.ts`. |

**UI:** при выделении своего Palace в сайдбаре появляется панель `#palace-panel` (CSS в `index.html`) с прогресс-баром и кнопкой активации. Логика — `ui/palacePanel.ts`. Для Death Hand кнопка переходит в режим targeting (красный фон), любой ЛКМ по миру запускает удар. ПКМ отменяет targeting.

**AI:** в `updateAI` AI строит Palace когда есть `hitech + yardupg + 1100 spice`. Активация супер-способности происходит сразу как только она готова — Harkonnen целится в CY игрока, остальные не требуют таргетинга.

Тонкие моменты:
- `superCooldown` инициализируется в `makeBuilding` только для `kind === 'palace'` (300с при постройке).
- `tickPalaces(state, dt)` в главном цикле уменьшает кулдаун у всех живых, не строящихся Palace-зданий.
- Death Hand-снаряд использует общий splash-урон в `updateProjectiles`. Не повреждает свою фракцию (стандартное правило). `targetId` = -1 — бьёт по точке, не по entity.
- Fremen-юнит не имеет фабрики — попытка `aiQueueUnit('fremen', ...)` вернёт false через `canFactionProduce`. Используется только через Palace.
- При установленном `palaceTargetingId` мышь блокирует выделение/перемещение — обрабатывает только targeting.

### Launcher — точность
Поле `accuracy` на юните (0–1). Для `launcher` точность зависит от дистанции:
- < 3 тайлов: 20% (стрельба вблизи нестабильна)
- 3–6.5+ тайлов: линейно растёт до 85%

При промахе (random > accuracy) к позиции цели добавляется случайный scatter, пропорциональный `(1 - acc) * 4`. Splash всё равно наносит урон по области рядом с целью.

## Комбайн (Harvester) и орнитоптер (Carryall)

**Harvester FSM:** `idle → toSpice → mining → toRefinery → (depositing) → idle`  
Дополнительное состояние `moving` — ручной приказ, не сбрасывается авто-поиском.

### Поведение харвестера и туман войны
Игрок видит спайс **только на открытых тайлах** (`fog ≥ 1`) — авто-поиск спайса для харвестера игрока ограничен туманом войны. AI игнорирует туман (видит весь спайс на карте).

Логика принятия решений (`systems/harvester.ts`, переменная `respectFog = unit.faction === state.faction`):
- **`idle`** + полный груз (≥ 500) → сразу в `toRefinery`.
- **`idle`** + есть открытый спайс на карте → `toSpice` к ближайшему.
- **`idle`** + нет видимого спайса, но есть частичный груз → `toRefinery` (не таскать впустую).
- **`idle`** + пусто и спайса не видно → ждёт на месте (игроку нужно скаутить новые поля).
- **`mining`** + тайл исчерпан, груз < 500, спайс на карте видим → `toSpice` к ближайшему (продолжает добычу!).
- **`mining`** + тайл исчерпан, груз ≥ 500 → возврат в `toRefinery`.
- **`mining`** + тайл исчерпан, груз 1–499 и спайса больше не видно → возврат в `toRefinery`.
- **`mining`** + тайл исчерпан, пусто и спайса не видно → `idle`.

**Ключевой принцип:** харвестер не возвращается на рефинерию пока не полон ИЛИ пока на видимой карте есть спайс. Это убирает «мотание туда-сюда» с половинным грузом, которое теряет эффективность.

Утилита `findNearestSpice(state, x, y, restrictToFog)` принимает опциональный флаг — `true` для игрока, `false` (по умолчанию) для AI. Утилита `hasKnownSpice(state, forPlayer)` отвечает «есть ли вообще видимый спайс».

### Ручной приказ ехать на рефинерию (`Unit.commandedReturn`)
Клик ПКМ по своей рефинерии (`mouse.ts`) переводит харвестер в `toRefinery` **и ставит флаг `commandedReturn = true`** независимо от груза. Полезно отозвать частично загруженный харвестер до атаки червя или рейда.

Семантика флага:
- Любой другой ручной приказ (`u.kind === 'harvester' && клик по точке`) сбрасывает флаг (`commandedReturn = false`, `mode = 'moving'`).
- Флаг сбрасывается автоматически после успешной выгрузки в рефинерии (`depositing → idle`) — авто-добыча возобновляется штатно.
- В авто-шаттле карриолла харвестер с `commandedReturn` идёт **с приоритетом**: подбирается независимо от расстояния (`groundDist < AUTO_CARRY_DIST` не отменяет подбор), и `score += 1000` ставит его впереди очереди. Без флага carryall работает по старой схеме (только дистанция > 6 тайлов).

### Carryall Auto-shuttle
По умолчанию `autoCarry = true`. Подбирает комбайн, если расстояние до цели по земле > 6 тайлов.  
Ручное управление (ПКМ) переводит в `autoCarry = false`. Через 2.5с простоя без активной задачи — возврат.

### Ручное управление карриоллом — двухшаговая команда
Каждый ПКМ задаёт ОДНО намерение, не отменяя другое:
- **ПКМ по своему харвестеру (без груза)** → `carryTarget = harvester.id`, `target = null`. Идёт подбирать. Старый loiter-target стирается.
- **ПКМ по своей рефинерии** → `target = (dock entry)`, `carryTarget` сохраняется. Снэп к точке дока (правый нижний угол рефинерии).
- **ПКМ по тайлу со спайсом** → `target = (tile centre)`, `carryTarget` сохраняется. Снэп к центру тайла, чтобы `dropCargo` попал на спайс.
- **ПКМ по любой другой точке** → `target = (wx, wy)`, `carryTarget` сохраняется.

Это даёт игроку команду «забрать X и доставить в Y» через два клика в любом порядке, кроме случая когда destination выставляется ПЕРЕД pickup — тогда destination стирается на pickup-клике (стирается auto-loiter target после предыдущей доставки).

Логика обработки в `updateCarryall` (порядок приоритета): cargo → carryTarget → target → idle.  
`pickUp()` ставит автоматический destination ТОЛЬКО если `c.target == null` — игроком заданная цель не перезаписывается.

### После выгрузки (`dropCargo`)
- `lastDropId` + `dropCooldown = 3.0с` блокируют повторный pickup того же юнита (иначе цикл «подобрал → выгрузил» вызывает дрожание спрайтов).
- Карриолл получает явный `c.target` = loiter-точка над рефинерией и улетает на полной скорости (`SPEED * dt`), не зависая над харвестером.
- Если выгрузка на спайс-тайл и груз пустой — харвестеру сразу ставится `mode = 'mining'` со снэпом позиции в центр тайла, чтобы `Math.floor(x)` гарантированно попал на спайс.
- При поиске нового спайса для авто-шаттла учитывается `respectFog` для игрока.

**Sandworm:** комбайны на песке могут быть съедены червём (см. секцию выше). Безопасные клетки — `rock`. Docked-харвестер (внутри рефинерии) неуязвим для червя/сплеша/picking. Стратегия: размещай рафинерии у каменистых плато.

## AI Brain

Цикл каждые 4–7 секунд:
1. Если `power.prod ≤ power.cons + 2` → строить Wind Trap
2. Пройти тех-дерево снизу вверх → строить первое недостающее здание
3. Апгрейды: `heavyupg` после Heavy Factory; `yardupg` после Radar
4. Защитный слой: до 2 Gun Turret после Radar; до 2 Rocket Turret после `yardupg`
5. Дублировать рафинерии для масштаба
6. Тренировать юниты (микс по доступным фабрикам, включая launcher при наличии heavyupg)
7. Скаутить быстрыми юнитами (`scoutTimer`)
8. Отправлять волны атаки (`attackTimer`, размер растёт со временем). Юнитам с активным `attackTarget` приказ марша **не** перевыдаётся — они доводят бой и сами возвращаются к движению.
9. **Continuous threat response** — каждый тик AI-мозг сканирует свои простаивающие боевые юниты и автоматически наводит их на ближайшего врага в радиусе 8 тайлов. Это убирает баг, когда волна шла к базе игрока и игнорировала любые атаки по пути.

### Возмездие за урон
`damageEntity(state, ent, dmg, attacker)` принимает атакующего. Если жертва-юнит без текущей цели и атакующий тоже юнит — жертва автоматически наводится на стрелка. Поле `Projectile.sourceId` хранит id атакующего, проставляется в обоих `fireProjectile` и `fireBuildingProjectile`.

### Авто-таргетинг во время движения
В главном цикле `main.ts` юнит, идущий по приказу (`u.target` set), всё равно сканирует ближайших врагов в радиусе `range × 1.1`. Идущий «без цели» юнит — в радиусе `range × 1.5`. После уничтожения цели `u.target` сохраняется → юнит возвращается на маршрут.

### `Unit.holdFire` — приоритет ручных приказов
Если игрок выдаёт юниту команду движения (ПКМ по точке), ставится `u.holdFire = true`. В этом состоянии авто-таргетинг **не активируется** — юнит игнорирует врагов и идёт куда сказано. Флаг сбрасывается, как только `u.target` становится `null` (юнит дошёл).

ПКМ по врагу (атакующая команда) — `u.holdFire = false` и `attackTarget = enemyId`. Юнит атакует и принимает авто-таргетинг.

Это убирает баг «юнит не реагирует на приказы пока стреляет»: новая команда сразу прерывает текущую стрельбу/преследование, потому что `attackTarget = null` сбрасывает старую цель, а `holdFire` не даёт авто-таргетингу её сразу же восстановить.

## Соглашения по коду

- **Стиль:** `camelCase` переменные, `PascalCase` типы/интерфейсы, `UPPER_SNAKE` константы
- **Строки:** 100 символов максимум
- **Отступ:** 2 пробела
- **Типы:** без `any`, использовать `unknown` при необходимости
- **Импорты:** относительные пути (`./ ../`), без алиасов — упрощает навигацию
- **Мутация стейта:** только внутри `systems/` и `state/`, рендер только читает
- **Комментарии:** на русском для логики/дизайна-решений, на английском для JSDoc

### Запрещено
- Глобальные переменные кроме `game: GameState | null` в `state/gameState.ts`
- Классы с наследованием (использовать интерфейсы + фабрики)
- Прямое обращение к DOM из `systems/` или `render/`
- `setTimeout` или `setInterval` для игровой логики (только `app.ticker`)
- `UNIT_STATS[kind]` напрямую для боевой/построечной логики — только `statsFor(faction, kind)`
